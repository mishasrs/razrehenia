# Разрешения

## What This Is

Это Google Apps Script-приложение, привязанное к Google Sheets, для ведения реестра разрешений из PDF-файлов в Google Drive. Скрипт сканирует папки, извлекает данные из имени файла и QR-кода, собирает рабочую витрину для оператора и поддерживает отправку документов через Telegram.

## Core Value

Оператор должен получать актуальный и надёжный реестр разрешений из Drive без ручной пересборки данных и без потери рабочих сценариев отправки.

## Requirements

### Validated

- ✓ Пользователь может указать корневую папку Google Drive для источника PDF-разрешений — existing
- ✓ Пользователь может запустить сканирование и синхронизацию реестра разрешений из Drive в Google Sheets — existing
- ✓ Система может дозаполнять отсутствующие QR-значения по уже загруженным PDF — existing
- ✓ Система поддерживает витрину по разрешениям для операторской работы в отдельном листе — existing
- ✓ Пользователь может диагностировать QR-распознавание по одному PDF-файлу — existing
- ✓ Система поддерживает Telegram-интеграцию для массовой отправки выбранных разрешений — existing
- ✓ Система поддерживает Telegram-бота для сотрудников с ролями, контактами и папками по ролям — existing

### Active

- [ ] Проверить текущие сценарии сканирования, синхронизации, QR-распознавания и Telegram-отправки на работоспособность и зафиксировать реальные риски
- [ ] Устранить дефекты и хрупкие места, которые мешают стабильной ежедневной работе оператора
- [ ] Оптимизировать узкие места по производительности и сопровождению без поломки текущих пользовательских сценариев
- [ ] Подготовить репозиторий к открытому публичному использованию на GitHub без утечки секретов и без скрытых предположений о среде

### Out of Scope

- Полная миграция с Google Apps Script на другой runtime — слишком большой объём относительно текущей задачи стабилизации
- Крупная продуктовая переработка пользовательского интерфейса — сейчас приоритет на надёжность существующего процесса
- Добавление новых каналов доставки помимо Google Sheets, Drive и Telegram — не требуется для текущего цикла улучшений

## Context

Проект уже существует как рабочая brownfield-кодовая база в корне репозитория и синхронизируется через `clasp`. Основной поток завязан на `00_MainUi.gs`, `40_ScanJob.gs`, `11_SheetsRepo.gs`, `12_QrService.gs`, `20_View.gs`, `31_Telegram.gs` и `90_Utils.gs`.

Код уже реализует сканирование дерева папок Google Drive, синхронизацию канонического листа разрешений, построение витрины, извлечение QR через внешние API и Telegram-рассылку. При этом в проекте нет автоматизированных тестов, локального test harness, CI и линтера, поэтому проверка работоспособности сейчас возможна только через статический аудит, синтаксическую проверку и ручные сценарии в среде Apps Script.

Репозиторий был переведён в публичный режим на GitHub, поэтому теперь особенно важно убрать неоднозначности в документации, проверить границы конфигурации и не допустить публикации секретов или чувствительных operational assumptions.

## Constraints

- **Tech stack**: Google Apps Script + Google Sheets + Google Drive + Telegram Bot API — проект уже построен вокруг сервисов Google Workspace и не должен ломать этот контур
- **Execution model**: Ограничения квот и времени Apps Script — длинные операции должны оставаться chunked и продолжаться через triggers
- **Verification**: Нет автоматизированных тестов и локальной эмуляции Apps Script — работоспособность нужно подтверждать через аудит, точечные проверки и ручные сценарии
- **Security**: Репозиторий теперь публичный — конфигурация должна оставаться в `PropertiesService`, а документация и код не должны раскрывать секреты
- **Compatibility**: Нельзя ломать существующие сценарии оператора в листах `Разрешения`, `Витрина`, `Logs` и Telegram-потоках — это уже используемые рабочие поверхности
