# Соглашения

## Область действия
- Этот репозиторий представляет собой bound-проект Google Apps Script без сборки и без локального модульного загрузчика.
- Рабочий код лежит в корне репозитория в файлах `*.gs`, `TelegramSend.html` и `appsscript.json`.
- Файл `.claspignore` сейчас пропускает к деплою только `*.gs`, `*.html` и `appsscript.json`, поэтому новый runtime-код нельзя прятать в подпапки без осознанного изменения sync-правил.
- `.clasp.json` использует `rootDir: ""` и пустой `filePushOrder`, поэтому фактическая организация проекта держится на нумерации файлов и глобальном namespace Apps Script.

## Именование файлов и порядок
- Файлы используют числовые префиксы как часть архитектурного контракта: `00_MainUi.gs` для UI-входов, `01_Config.gs` для общих констант, `02_RunControl.gs` и `03_Logging.gs` для инфраструктуры, `10_*` и `11_*` для репозиториев, `12_QrService.gs` для внешнего QR-пайплайна, `20_*` для листов-представлений, `31_Telegram.gs` для интеграции Telegram, `40_ScanJob.gs` для orchestration, `90_Utils.gs` для чистых helper-функций.
- Если появляется новый модуль, его лучше встраивать в этот диапазон, а не создавать общий файл вроде `helpers.gs` или `misc.gs`.
- HTML-ресурс называется по фактическому серверному include: `HtmlService.createHtmlOutputFromFile('TelegramSend')` в `31_Telegram.gs` ожидает файл `TelegramSend.html` с тем же именем.
- Практический пример: новый sidebar для отдельного сценария стоит оформлять как отдельный HTML-файл в корне и открывать его по имени, по образцу `TelegramSend.html`.

## Именование функций и область видимости
- Публичные entry points не имеют завершающего подчёркивания, потому что вызываются из меню, триггеров или `google.script.run`: `onOpen`, `scanAndUpdate`, `continueScan`, `fillMissingQr`, `diagnoseOne`, `refreshView`, `tgOpenSendSidebar`, `tgGetSidebarData`, `tgSendFromUi`.
- Внутренние helpers помечаются завершающим `_`: `ensureSheet_`, `scheduleContinue_`, `parsePermissionFilename_`, `decodeQrFromPdf_`, `tgNormalizeRole_`, `tgApplyEmployeeBotCommands_`.
- Для подсистем используется доменный префикс. В `31_Telegram.gs` почти все функции и константы начинаются с `tg`, что защищает глобальный namespace Apps Script от случайных конфликтов.
- Частый паттерн качества: тонкая публичная оболочка плюс internal-core. Пример: `scanAndUpdate()` вызывает `scanAndUpdateCore_('manual')`, а `continueScan()` вызывает тот же core с другим origin в `40_ScanJob.gs`.
- Если новая функция должна быть доступна пользователю через меню или триггер, её имя должно оставаться стабильным и без `_`, иначе Apps Script не сможет вызвать её напрямую.

## Стиль кода и структура функций
- Кодовая база написана процедурно, без классов. Состояние передаётся через аргументы, листы, `PropertiesService` и локальные объекты.
- По умолчанию используется `const`. `let` применяется только при реальном изменении значения, например `cursor` в `11_SheetsRepo.gs`, `added` в `40_ScanJob.gs` или `thumbUrl` в `12_QrService.gs`.
- Важная норма стиля здесь: ранние выходы вместо глубокой вложенности. Это видно в `uiSetRootFolder` из `00_MainUi.gs`, `tgOpenSendSidebar` и `tgSendFromUi` из `31_Telegram.gs`, `parsePermissionFilename_` из `90_Utils.gs`.
- Большие файлы размечаются секциями через banner-комментарии `/** ===================== ... ===================== */`. Этот способ уже используется в `00_MainUi.gs`, `12_QrService.gs`, `31_Telegram.gs` и `40_ScanJob.gs`.
- Комментарии описывают контракт и доменное поведение, а не синтаксис. Самые полезные примеры находятся в шапке `90_Utils.gs` и в пояснениях к синхронизации в `11_SheetsRepo.gs`.
- Практический пример: если добавляется новая chunked-операция, её лучше оформлять по образцу `syncExistingRowsChunked_` или `fillMissingQr`, с явными счётчиками, early return и логом результата.

## Именование данных и объектных структур
- Константы верхнего уровня пишутся в `UPPER_SNAKE_CASE`: `SHEET_NAME`, `LOG_SHEET_NAME`, `TIME_BUDGET_MS`, `FILL_MISSING_LIMIT`, `TG_THROTTLE_MS`.
- Ключи Script Properties поднимаются как `PROP_*`: `PROP_STOP_SCAN`, `PROP_SYNC_CURSOR`, `PROP_TG_TOKEN`, `PROP_TG_UPD_OFFSET`, `PROP_TG_DRIVER_FOLDER_ID`.
- Индексы колонок оформлены как 1-based `COL_*` в `01_Config.gs`, потому что Sheets API работает с 1-based диапазонами. Магические номера в коде считаются отклонением от текущей нормы.
- Поля временных DTO называются коротко и доменно: `ts`, `route`, `tonnage`, `width`, `until`, `fileUrl`, `qrUrl`, `fileId`. Эти же поля проходят через `20_View.gs`, `31_Telegram.gs` и `90_Utils.gs`.
- Тексты UI и пользовательские сообщения в коде в основном русские, а технические идентификаторы остаются английскими. Это устойчивая смешанная норма проекта.
- Практический пример: если добавляется новая Telegram-роль, нужно сохранить текущую схему `TG_ROLE_*`, `PROP_TG_*_FOLDER_ID` и нормализацию через `tgNormalizeRole_`, а не вводить строковые литералы по месту.

## Константы и конфигурация
- Глобальная общепроектная конфигурация сосредоточена в `01_Config.gs`.
- Подсистемные константы остаются рядом со своей областью ответственности. `TG_CONTACTS_HEADERS`, `TG_EMPLOYEE_COMMANDS` и `TG_BTN_*` живут в `31_Telegram.gs`, а не в `01_Config.gs`.
- Изменяемые значения окружения хранятся не в коде, а в `PropertiesService.getScriptProperties()`. Это касается `ROOT_FOLDER_ID`, Telegram token и опциональных folder override для ролей.
- `appsscript.json` хранит только runtime-настройки платформы: `runtimeVersion: "V8"`, `exceptionLogging: "STACKDRIVER"` и advanced service `Drive v2`.
- Практический пример: если появляется новый флаг поведения, сначала нужно решить, это compile-time константа уровня `DEBUG` в `01_Config.gs` или runtime-настройка пользователя уровня `ROOT_FOLDER_ID`/`TELEGRAM_BOT_TOKEN`.
- При изменении структуры листа недостаточно поправить только заголовок. Нужно синхронно обновлять `COL_*` в `01_Config.gs`, создание заголовков в `11_SheetsRepo.gs`, чтение строк в `20_View.gs` и выборки в `31_Telegram.gs`.

## Логи, исключения и уровень строгости
- Главный диагностический канал проекта сейчас не Stackdriver, а лист `Logs`, который создаёт `ensureLogSheet_` и заполняет `log_` из `03_Logging.gs`.
- UI-обратная связь должна быть короткой и неблокирующей через `toastSafe_`, а подробности ошибки должны попадать в лог.
- Для инфраструктурных и вторичных операций проект допускает пустые `catch`-блоки. Примеры: `onOpen` в `00_MainUi.gs`, `log_` и `toastSafe_` в `03_Logging.gs`, несколько fallback-веток в `12_QrService.gs`.
- Для пользовательских и интеграционных boundary-функций принят более строгий подход: явно проверять ответ и бросать `Error` с контекстом. Это видно в `getRootFolderId_`, `tgApplyEmployeeBotCommands_`, `tgSyncChatIdsFromUpdates`, `tgSendMessage_`, `tgSendDocument_`.
- Сетевые вызовы выполняются через `UrlFetchApp.fetch` с `muteHttpExceptions: true`, после чего код сам валидирует `HTTP` и полезную нагрузку. Такой шаблон стоит сохранять и для новых интеграций.
- Лог-сообщения используют короткие уровни вроде `INFO`, `WARN`, `ERROR`, `DEBUG`. Если добавляется новый лог, лучше оставаться в этой шкале.
- Практический пример: новый HTTP-helper должен выглядеть ближе к `tgSendMessage_`, чем к silent helper'ам QR-ветки, потому что пользовательский эффект там критичен.

## Идиомы Apps Script
- Идемпотентные `ensure*`-helpers считаются базовым стилем работы с таблицами и листами: `ensureLogSheet_`, `ensureSheet_`, `ensureViewSheet_`, `tgEnsureContactsSheet_`.
- Чтение и запись данных по возможности батчатся. `11_SheetsRepo.gs` сначала получает `getValues()` и `getRichTextValues()`, меняет матрицы в памяти, затем делает `setValues()` и `setRichTextValues()`.
- Гиперссылки в таблицах хранятся как `RichTextValue`, а не как сырые URL в видимом тексте. Это видно в `appendRow_`, `syncExistingRowsChunked_`, `makeLink_`, `writeSectionGroupedByTs_`.
- Долгие задачи защищены `LockService.getScriptLock()` и anti-timeout логикой из `02_RunControl.gs`. Повторяющиеся триггеры всегда очищаются перед созданием нового.
- Внешний UI для sidebar написан на ванильном HTML/JS без сборки. Обмен с сервером идёт через `google.script.run` в `TelegramSend.html`.
- Drive- и Telegram-интеграции пишутся с fallback-поведением и паузами `Utilities.sleep(...)`, что соответствует ограничениям платформы и внешних API.

## Практические паттерны, которые стоит сохранять
- Для чистой логики парсинга и форматирования придерживайтесь стиля `90_Utils.gs`: маленькие функции, строковый вход, простой объект или scalar на выходе, без побочных эффектов.
- Для orchestration-функций придерживайтесь стиля `40_ScanJob.gs`: взять lock, прочитать контекст, выполнить пачку работы, залогировать результат, безопасно завершиться или спланировать продолжение.
- Для табличных обновлений придерживайтесь стиля `11_SheetsRepo.gs`: работать через массивы строк и константы колонок, а не через россыпь одиночных `getRange(...).setValue(...)`.
- Для Telegram-флоу придерживайтесь стиля `31_Telegram.gs`: сначала проверить обязательные настройки, затем нормализовать входные данные, затем выполнить сетевой вызов, затем вернуть компактный результат и залогировать сбои.
- Для новых UI-действий меню придерживайтесь стиля `00_MainUi.gs`: отдельная функция на действие, короткий `prompt`/`alert`, без дублирования бизнес-логики прямо в menu-handler.
