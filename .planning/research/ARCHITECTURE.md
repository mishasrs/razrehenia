# Архитектурное исследование

## Цель

Нужно стабилизировать и ускорить существующий Google Apps Script-проект без смены платформы. Для этого системе не нужен новый runtime; ей нужен более дисциплинированный модульный монолит внутри текущего Apps Script-контура, где:

- пользовательские точки входа остаются совместимыми с меню и триггерами;
- каноническое хранилище остаётся в Google Sheets;
- тяжёлые и хрупкие внешние шаги отделяются от критического пути;
- конфигурация, секреты, курсоры и интеграционные границы становятся явными.

## Что уже есть в репозитории

- `00_MainUi.gs` публикует пользовательские entry points через меню таблицы.
- `40_ScanJob.gs` координирует основной scan/sync flow и дозаполнение QR.
- `11_SheetsRepo.gs` хранит канонический реестр разрешений в листе `Разрешения`.
- `20_View.gs` строит операторскую витрину `Витрина` как производную проекцию.
- `12_QrService.gs` извлекает QR через Drive thumbnail и внешние HTTP-сервисы.
- `31_Telegram.gs` объединяет sidebar, контакты, роли, bot polling и отправку документов.
- `01_Config.gs` и `PropertiesService` уже используются как конфигурационная граница.
- `appsscript.json` уже работает на `V8` и использует advanced service `Drive v2`.

Текущий код уже близок к модульному монолиту, но границы местами размыты:

- `11_SheetsRepo.gs` не только пишет в Sheets, но и сам ходит в `DriveApp`.
- `31_Telegram.gs` смешивает UI, application-логику, хранение контактов и транспорт к Telegram API.
- QR-распознавание встроено в путь первичного добавления новых PDF и замедляет критический путь.
- витрина и Telegram знают о скрытых колонках `H:I` как о неформальном контракте.

## Рекомендуемая целевая форма

Рекомендуемая форма для этого проекта: **modular monolith on Apps Script**, а не микросервисы и не перенос в другой runtime. В Apps Script нет import/export-границ, поэтому границы нужно держать файлами, именованием и стабильными публичными entry points.

### Рекомендуемые границы

| Граница | Ответственность | Текущее состояние | Рекомендация |
|---|---|---|---|
| UI / entry points | Меню, prompt/alert, sidebar boot | `00_MainUi.gs`, `TelegramSend.html`, часть `31_Telegram.gs` | Оставить тонким слоем без бизнес-решений |
| Application orchestration | Сценарии `scan`, `continue`, `fillMissingQr`, Telegram use cases | `40_ScanJob.gs`, часть `31_Telegram.gs` | Здесь должен жить только порядок шагов, lock, retries, progress |
| Domain rules | парсинг имени, нормализация роли, статусы записи, правила маршрута | `90_Utils.gs`, часть `31_Telegram.gs`, часть `12_QrService.gs` | Вытащить в чистые функции с детерминированным результатом |
| Repositories / adapters | Drive, Sheets, QR HTTP, Telegram HTTP, Properties, Cache | `10_DriveRepo.gs`, `11_SheetsRepo.gs`, `12_QrService.gs`, часть `31_Telegram.gs` | Каждый adapter отвечает только за свой внешний сервис |
| Projections | построение `Витрина`, форматирование, скрытые техполя | `20_View.gs`, `21_DataFormat.gs` | Считать витрину только производной проекцией |
| Observability | логи, execution id, диагностика, статус последнего прогона | `03_Logging.gs`, часть `02_RunControl.gs` | Разделить операторский лог и инженерную диагностику |
| Config / secrets | константы схемы, флаги, свойства окружения, токены | `01_Config.gs`, `PropertiesService` | Хранить только компактное конфигурационное и курсорное состояние |

### Практическое следствие для этого репозитория

- Публичные функции `scanAndUpdate`, `continueScan`, `fillMissingQr`, `refreshView`, `tgOpenSendSidebar`, `tgSendFromUi`, `tgProcessEmployeeBotUpdates` должны остаться стабильными.
- Внутреннюю декомпозицию безопаснее делать новыми `*.gs` файлами по текущей номерной схеме, а не менять платформу или способ деплоя.
- Самый полезный разрез для brownfield-стабилизации сейчас не по техническим слоям Google, а по пользовательским потокам: `scan`, `qr enrichment`, `view projection`, `telegram delivery`.

## Каким должен быть очищенный поток данных

### 1. Discovery и инвентаризация файлов

Поток `Drive -> scan` должен сначала получать только факты о файле:

- `fileId`
- имя
- URL
- принадлежность дереву папок
- состояние доступа
- признак `trashed`

Это логически зона `10_DriveRepo.gs`. `11_SheetsRepo.gs` не должен сам принимать решение о существовании файла через `DriveApp.getFileById(...)`; сейчас это смешивает persistence и внешний I/O.

### 2. Нормализация и решение по записи

После discovery нужен отдельный шаг решения:

- файл валиден и парсится;
- файл валиден, но имя не распознано;
- файл временно недоступен;
- файл подтверждённо удалён или вышел из дерева;
- запись требует QR enrichment;
- запись готова к проекции и доставке.

Сейчас часть этих случаев схлопнута в `null`, silent skip или немедленное удаление строки. Для надёжности это нужно разделить. Минимально безопасный вариант без ломки основной схемы: завести отдельный служебный список проблемных файлов, а не расширять сразу лист `Разрешения` новыми рабочими колонками.

### 3. Канонический реестр

Лист `Разрешения` должен оставаться system of record, но с более жёстким правилом:

- только `11_SheetsRepo.gs` пишет строки реестра;
- application-слой передаёт туда уже нормализованные изменения;
- удаление строки допускается только после подтверждённого состояния `deleted/trashed/out_of_scope`, а не при любой ошибке чтения Drive;
- QR не должен быть обязательным для появления строки в реестре.

Ключевой архитектурный сдвиг для оптимизации: **первичное сканирование и QR enrichment должны быть разнесены**. Сейчас `40_ScanJob.gs` вызывает `decodeQrFromPdf_()` в цикле добавления новых PDF. Это делает внешний `UrlFetchApp` и image processing частью критического пути, хотя QR нужен не для существования записи, а для её обогащения.

Рекомендуемый поток:

1. Scan sync создаёт или обновляет запись без ожидания QR.
2. Отдельный enrichment step дозаполняет QR пачками.
3. Ошибки QR не мешают консистентности основного реестра.

Это лучше соответствует ограничениям Apps Script по времени выполнения и квотам на `UrlFetchApp`.

### 4. Проекции и пользовательские экраны

`20_View.gs` должен остаться чистым projection builder:

- входом служат только канонические строки;
- `Витрина` не редактируется как источник истины;
- скрытые технические поля остаются допустимыми, но их схема должна быть централизована в одном месте, а не дублироваться между `20_View.gs` и `31_Telegram.gs`;
- отправка из витрины должна опираться не на доверие к HTML payload, а на повторную серверную materialization по `fileId` и допустимым строкам.

### 5. Доставка через Telegram

Для `31_Telegram.gs` правильная граница такая:

- контакты и роли: отдельный adapter над листом `Контакты`;
- bot polling и обработка команд: application flow;
- низкоуровневые `sendMessage` / `sendDocument`: transport adapter;
- sidebar: UI adapter;
- валидация payload и прав на отправку: server-side application logic.

Сейчас `tgSendFromUi(payload)` доверяет `chatIds` и `files`, пришедшим из HTML. Для public GitHub readiness и общей надёжности сервер должен заново проверять:

- что `chat_id` существует и активен;
- что роль и папка разрешены;
- что `fileId` принадлежит допустимой записи;
- что пользовательский кэш не превратился в устаревшее состояние.

`CacheService.getUserCache()` здесь допустим только как краткоживущее UI-удобство, но не как источник истины.

## Что стоит сделать с инфраструктурными границами

### Lock и конкурентность

- `scan` и `fillMissingQr` уже используют `LockService.getScriptLock()`, это правильная база.
- Telegram polling в `tgSyncChatIdsFromUpdates()` тоже должен работать под lock вокруг чтения и обновления `PROP_TG_UPD_OFFSET`, иначе параллельные запуски могут дублировать или перескакивать updates.
- Возврат по `!tryLock(...)` без лога для пользовательских сценариев слишком хрупок; нужен явный операционный сигнал в `Logs` и в execution logs.

### Triggers

- Продолжение длинных задач через time-driven trigger соответствует платформе и ограничению `6 min / execution`.
- Триггерную модель не нужно менять на webhook-first или внешний scheduler, пока текущий объём укладывается в Apps Script.
- Но trigger lifecycle должен быть наблюдаемым: кто создал, что продолжает, какой cursor, почему остановился.

Важно учитывать ограничение installable triggers: они работают от имени создателя триггера. Для этого проекта это означает, что служебные триггеры должны создаваться из технически контролируемой учётной записи, а не из случайной пользовательской сессии.

### Properties и runtime state

`PropertiesService` подходит для:

- `ROOT_FOLDER_ID`
- Telegram token и folder overrides
- stop-флага
- cursor и offset
- компактного статуса последнего прогона

`PropertiesService` не подходит как хранилище большого job-state или длинных JSON-слепков: сервис хранит строки и живёт в quota-модели Apps Script. Поэтому всё объёмное и пользовательское должно оставаться в Sheets или пересчитываться из источника.

### Логирование и диагностика

Текущий лист `Logs` полезен как операторский журнал, но его недостаточно для инженерной диагностики:

- он хрупок к очистке;
- пишет по одной строке;
- не даёт нормальной фильтрации по execution;
- не заменяет Cloud Logging.

Для текущего цикла достаточно двухуровневой модели:

- `Logs` как короткий операторский журнал;
- Cloud Logging как инженерный лог выполнения.

Для этого проекту полезно перейти на standard Google Cloud project, потому что именно он нужен для полноценного просмотра Cloud Logging и Error Reporting. Это соответствует задаче public GitHub readiness лучше, чем добавление новой платформы.

## Безопасный порядок рефакторинга

1. **Зафиксировать контракты без изменения поведения.** Описать схемы листов, скрытые колонки витрины, ключи properties и публичные entry points. Это снижает риск случайной регрессии при дальнейшем делении файлов.
2. **Усилить наблюдаемость.** Добавить явный лог для lock contention, start/finish execution id, состояние cursor/offset и причины остановки в `02_RunControl.gs`, `03_Logging.gs`, `40_ScanJob.gs`, `31_Telegram.gs`.
3. **Разделить чистую доменную логику и side effects.** Вынести результат парсинга имени, статусы синхронизации и нормализацию Telegram-ролей в чистые функции с предсказуемыми return-объектами вместо `null` и неструктурированных строк.
4. **Развести критический путь scan и QR enrichment.** Сначала добиваться надёжной синхронизации PDF в `Разрешения`, потом отдельным шагом дозаполнять QR. Это даст самый заметный выигрыш по времени и по локализации сбоев.
5. **Исправить границу Drive <-> Sheets.** Решения о существовании и доступности файла должны приниматься до записи, а `11_SheetsRepo.gs` должен стать именно репозиторием таблицы.
6. **Разрезать `31_Telegram.gs` по подсценариям без смены публичных функций.** Минимальный безопасный разрез: contacts, bot polling, sidebar selection, send transport, role folder resolution.
7. **Только после этого оптимизировать массовые операции.** Батч-добавление строк, уменьшение `DriveApp.getFileById(...)` и `UrlFetchApp` в циклах, сокращение полных rebuild `Витрина`, кэширование краткоживущих метаданных в пределах одного запуска.
8. **Параллельно с предыдущими шагами подготовить публичную эксплуатацию.** Проверить secret boundary, подключить standard Cloud project, раздать дополнительные owner/IAM права и, если доступно в домене, перевести контейнерный файл и сам Apps Script-проект в Shared Drive.

## Что не нужно делать сейчас

- Не нужно переписывать проект с Google Apps Script на Cloud Run, Node.js или другой backend.
- Не нужно выносить каноническое хранилище из Google Sheets в отдельную БД только ради “правильной архитектуры”.
- Не нужно переводить весь проект на event sourcing, Pub/Sub, очереди или микросервисы.
- Не нужно делать крупный UI rewrite для `Витрина` или sidebar, пока не устранены риски в scan/sync/Telegram flows.
- Не нужно вводить bundler, framework для HTML UI или сложный frontend state management; для Apps Script UI это добавит стоимость, а не надёжность.
- Не нужно сейчас мигрировать bound-проект в standalone только потому, что standalone удобнее для collaboration. Это разумно обсуждать после стабилизации, если текущая связка Spreadsheet UI + `clasp` начнёт реально мешать.
- Не нужно менять polling Telegram-бота на webhook, если сначала не доказано, что основная проблема именно в polling-модели, а не в отсутствии lock и server-side validation.

## Итоговая рекомендация

Для этого репозитория правильная архитектурная цель на текущий milestone: **не новая система, а более строгая форма существующей**.

Практически это означает:

- сохранить Apps Script, Sheets, Drive и Telegram как текущий платформенный контур;
- оформить код как явный модульный монолит с жёсткими файлами-границами;
- оставить `Разрешения` единственным каноническим реестром, а `Витрина` только проекцией;
- вынести QR и Telegram из критического пути scan там, где это возможно;
- сначала стабилизировать контракты, наблюдаемость и validation, а уже потом делать точечную оптимизацию.

Такая структура лучше всего соответствует brownfield-ограничениям этого проекта: она повышает надёжность, уменьшает стоимость регрессий и не требует платформенного переписывания.
